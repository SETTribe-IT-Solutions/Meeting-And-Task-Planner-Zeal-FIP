Phase 1 — Core Security Fixes (8 Items)
⚠ Fix 1 · Task Ownership Validation (P0 — IDOR)

Problem: Any logged-in user could update any other user's task status by POST-ing an arbitrary task_id. No ownership check existed.

Fix: UpdateTaskStatusController.php now verifies the requesting user owns the task before allowing update. Checks both the legacy assigned_to column and the newer task_assignments table. Collectors and Organizers bypass the check (by design).

// Employee ownership check — both assignment paths
if ($role !== 'Collector' && $role !== 'Organizer') {
    $authorized = false;
    $authStmt = $conn->prepare("SELECT assigned_to FROM tasks WHERE id = ? LIMIT 1");
    $authStmt->bind_param("i", $taskId);
    $authStmt->execute();
    // ... also checks task_assignments table if it exists ...
    if (!$authorized) { http_response_code(403); redirect(...); }
}
⚠ Fix 2 · Task & Attendance Status Whitelists

Problem: No validation on the status POST field — arbitrary strings could be written to the database.

Fix: Strict in_array(..., true) whitelist in both controllers. Invalid values are rejected immediately.

$ALLOWED_STATUSES = ['Pending', 'In Progress', 'Completed', 'Cancelled'];
if (!in_array($status, $ALLOWED_STATUSES, true)) {
    $_SESSION['error'] = 'Invalid status value.';
    redirect('../modules/tasks/index.php');
}
⚠ Fix 3 · SQL Injection via String Interpolation

Problem: Dashboard (index.php) built 3 SQL queries using string interpolation with user-session values inserted directly. Vulnerable to SQL injection.

Fix: All 3 queries converted to MySQLi prepared statements with bind_param().

// Before (vulnerable):
$conn->query("SELECT COUNT(*) FROM meetings WHERE meeting_date >= '$today'");

// After (safe):
$stmt = $conn->prepare("SELECT COUNT(*) as total FROM meetings WHERE meeting_date >= ? AND status != 'Cancelled'");
$stmt->bind_param("s", $today);
$stmt->execute();
🔒 Fix 4 · Password Migration — Plain-text → bcrypt (Transparent)

Problem: All seed/legacy accounts stored plain-text passwords. Forcing a reset would lock out all existing users.

Fix: On login, detect plain-text vs bcrypt via prefix check. If plain-text matches, authenticate AND immediately upgrade to bcrypt in one transaction. Completely transparent to the user — no forced reset, no notification, no disruption.

$isHashed = (bool) preg_match('/^\$2[ayb]\$/', $storedPassword);
if ($isHashed) {
    $passwordValid = password_verify($password, $storedPassword);
} elseif ($password === $storedPassword) {
    $passwordValid = true;
    $newHash = password_hash($password, PASSWORD_BCRYPT);
    $upgradeStmt = $conn->prepare("UPDATE users SET password = ? WHERE id = ?");
    $upgradeStmt->bind_param("si", $newHash, $user['id']);
    $upgradeStmt->execute(); // silently upgrade
}
⚠ Fix 5 · Self-Registration Privilege Escalation

Problem: Anyone could self-register as Collector or Organizer by POST-ing role=Collector — gaining full admin access instantly.

Fix: RegisterController.php now whitelists only Employee for self-registration. Collector and Organizer accounts must be created by an admin through the Users module.

// Self-registration restricted to Employee only
private static $VALID_ROLES = ['Employee'];
⚠ Fix 6 · sanitizeInput() Double-Encoding Bug

Problem: sanitizeInput() applied htmlspecialchars() before DB storage. Views then applied it again at render time — ampersands became &amp;, quotes became &quot;. Data was corrupted in UI. Also broke redirect URLs (& became &amp;).

Fix: Removed htmlspecialchars() from sanitizeInput(). Escaping now happens only at the output/render layer, which is the correct architecture.

function sanitizeInput($data) {
    $data = trim($data);
    $data = stripslashes($data);
    return $data; // htmlspecialchars() removed — applied at render time only
}
⚠ Fix 7 · Debug Output Disabled in Production

Problem: debug() function dumped raw PHP data to the browser. No production gate existed.

Fix: Added APP_DEBUG constant (set to false). The debug() function returns immediately if APP_DEBUG is false.

define('APP_DEBUG', false); // set true only in local dev

function debug($data) {
    if (!APP_DEBUG) { return; } // silent in production
    echo "<pre>"; print_r($data); echo "</pre>";
}
⚠ Fix 8 · Open Redirect in AttendanceController

Problem: The redirect POST parameter was passed directly to header(Location:), allowing attackers to redirect users to external malicious sites after a form submission.

Fix: Whitelist regex validates the redirect path before use. Only relative paths matching ../modules/... are accepted.

if (!empty($_POST['redirect'])) {
    $r = trim($_POST['redirect']);
    if (preg_match('#^\.\./modules/[a-zA-Z0-9_/.\-?=&]+$#', $r)) {
        $redirectUrl = $r;
    }
}
Phase 1.1 — Security Hardening (4 Items)
🔒 Fix 9 · Meeting Role Validation Bug (Collectors Blocked)

Problem: MeetingController.php checked role !== 'Organizer' which silently blocked Collectors from creating meetings, even though Collectors are supposed to have full permissions.

Fix: Replaced with the existing isOrganizer() helper which correctly returns true for both Organizer and Collector roles.

// Before (broken — blocked Collectors):
if (!isset($_SESSION['user_id']) || $_SESSION['role'] !== 'Organizer') { ...

// After (correct):
if (!isset($_SESSION['user_id']) || !isOrganizer()) { ...
🔒 Fix 10 · CSRF Protection on All State-Changing Forms

Problem: AttendanceController.php and UpdateTaskStatusController.php had no CSRF protection — forged cross-site requests could change task statuses or attendance records without user knowledge.

Fix: Added hash_equals() CSRF token check at the top of every POST handler. Added hidden csrf_token inputs to all affected forms. Added initialization guard in config/db.php so existing sessions get a token without requiring re-login.

Forms updated: Task status (×2 forms), Meeting invite, Attendance update in meeting view (×N attendees), Attendance inline update.

// Controller side — every POST handler:
$submitted_token = trim($_POST['csrf_token'] ?? '');
if (empty($submitted_token) || !hash_equals($_SESSION['csrf_token'] ?? '', $submitted_token)) {
    $_SESSION['error'] = 'Invalid security token. Please refresh and try again.';
    redirect(...);
}

// Form side — every affected form:
<input type="hidden" name="csrf_token"
       value="<?php echo htmlspecialchars($_SESSION['csrf_token'] ?? ''); ?>">
🔒 Fix 11 · Login Rate Limiting (Session-Based)

Problem: No limit on login attempts — the application was vulnerable to credential stuffing and brute-force attacks with unlimited attempts.

Fix: Session-based rate limiting in AuthController.php. 5 consecutive failures → 15-minute lockout. Counter resets on success. No database or Redis required.

// Check at login start:
if (!empty($_SESSION['login_lockout_until']) && time() < $_SESSION['login_lockout_until']) {
    $remaining = ceil(($_SESSION['login_lockout_until'] - time()) / 60);
    $_SESSION['error'] = "Too many failed login attempts. Try again in {$remaining} minute(s).";
    exit();
}

// On success — clear counters:
unset($_SESSION['login_attempts'], $_SESSION['login_lockout_until']);
⚙ Fix 12 · Registration UI — Remove Privileged Roles from Dropdown

Problem: The registration form still showed Collector and Organizer in the role dropdown, which was misleading (the server already blocked these via whitelist). Could confuse users and invite tampering attempts.

Fix: Removed Collector and Organizer options from modules/users/register.php. Only Employee is now displayed, matching the server-side enforcement.
